What is AI-DLC?

The AI Development Life Cycle — a repeatable, governed way to ship software when AI agents do most of the typing.

01 • What is AI-DLC?

AI-DLC: the SDLC, re-drawn for agents

The AI Development Life Cycle keeps the discipline of the classic SDLC — requirements, design, build, review, release — but assumes the work is done by AI agents operating under human direction, not by a room full of engineers typing.

The problem it solves

  • Ungoverned AI coding is fast but unauditable — who decided what, and why?
  • One giant prompt cannot hold design, build and review without losing the thread.
  • Agents happily edit anything, anywhere — no lanes, no contracts, no self-review.
  • Output quality is invisible until a human reads every line.

What AI-DLC adds

  • Phased flow — intent → plan → design → build → review → release.
  • Named roles with clear responsibility and file scope.
  • Contracts first — the interface is agreed before code is written.
  • Gates & self-review at every hand-off, so quality is enforced, not hoped for.
AI-DLC is the process. Agent Squad is the construct that makes agents follow it. The board manager (ai-dlc-board-manager) is the project-management layer on top.

The phases: one canonical delivery pipeline

Every change flows through the same six states, with the board as the single source of truth. Spec-first at the front, human gates at the back — so provenance runs unbroken from plan to merge.

Inside “Agent Work”: the Agent Squad staging

The board owns the outer pipeline. When a ticket enters Agent Work, Agent Squad runs its own guaranteed staging:

Design → Build → Review

The design contract MUST be in main before any build starts — the contract-first hard rule.

Human-in-the-loop by design

  • Spec-first: nothing is built without a brief and a testable check.
  • Agent-orchestrated: agents do the work between the gates.
  • Human-governed: a person moves Spec Ready and accepts / merges.

Accepted is optional — teams can collapse it into Done. Lifecycle hooks (pre/post-implement, pre/post-pr) enforce quality at each seam.

SECTION 02

The AI-DLC plugin (board-manager)

The Product-Owner and Tech-Lead layer that drives your board through the pipeline — usable on its own.

03 • The AI-DLC plugin

ai-dlc-board-manager: run delivery on the board you already have

A Claude Code / Cowork plugin (v0.6.0) that gives an AI agent Product-Owner and Tech-Lead skills. It manages an existing board — it does not build or host one. Board-manager owns the board; Agent Squad owns the coding agent.

The problem it solves

  • Turning specs into a plan, and a plan into board issues, is slow, manual and error-prone.
  • Sprint planning by hand ignores the real dependency graph between features.
  • Nobody can trace a shipped change back to the spec that asked for it.

Five guiding principles

  • Spec / plan-driven — specs come first, always.
  • Multi-agent collaboration — specialist roles, not one bot.
  • Board as universal interface — the single source of truth.
  • Living provenance — plan → issue → branch/PR → review.
  • Human-in-the-loop gates — people approve the transitions.
“AI-DLC” here means AI-Driven Development Life Cycle — this team’s own spec-first, agent-orchestrated, human-governed methodology. Board backend today is GitHub Projects v2; Jira, Azure DevOps and Linear are on the roadmap behind a provider contract.

05 • Together or independent

Two plugins: use them together, or on their own

They are a designed pair with a clean ownership split — but neither forces the other on you.

AI-DLC PLUGIN ALONE

Pure PM / board tool

Run spec-to-plan, board-sync, sprint-plan, start-feature to drive a GitHub board from specs. No coding agent required.

Only finish-feature needs Agent Squad — it politely refuses without it.

BOTH TOGETHER

Full AI-DLC loop

Board-manager plans and moves tickets; Agent Squad designs, builds and reviews inside Agent Work. Provenance runs end-to-end: spec → issue → PR → merge.

The hand-off: start-feature → /agent-squad:implement → finish-feature.

AGENT SQUAD ALONE

Governed coding, no board

Run orchestrate / implement / finish-feature straight off a GitHub issue or a brief. Personas, lanes, self-review and token tracking all work with no board at all.

Board mirroring simply stays off.

Ownership, in one line: board-manager owns the board and the plan; Agent Squad owns the agents and the code. Wire them with a single .ai-dlc.yml when you want the full loop — or adopt just the half you need today and add the other later.

SECTION 03

What is Agent Squad?

The behaviour contract that turns raw AI coding into a governed engineering team.

03 • What is Agent Squad?

Agent Squad is a behaviour contract, not a bot

An organisation-wide construct that defines how AI engineering agents behave: three personas, a role-composition schema, a self-review format, and the lifecycle / tool hooks that enforce them. It ships as a Claude Code plugin (v0.7.0).

A

Deliberately narrow

It knows nothing about your board, your language, or your domain. That keeps it reusable across every project.

B

Layered, not monolithic

Behaviour, knowledge and file scope are separate layers — you compose them into project-specific roles.

C

Enforced, not advisory

Governance rules run as hooks. An agent that strays out of lane or skips self-review is blocked, not trusted.

Persona (behaviour)  +  Skills (knowledge)  +  Lanes (file scope)  =  Role

Example role: backend-dev = implementer + [python, fastapi, postgres] + writes only in app/api/, tests/

03 • The three-layer model

Persona + Skills + Lanes = Role

LAYER 1

Persona — behaviour

Fixed set of three (Lead, Architect, Implementer). Ships with the construct. Defines how an agent works — what it may and may not do.

LAYER 2

Skills — knowledge

Pluggable SKILL.md files (aws, docker, fastapi, postgres, python, react, terraform, typescript, github-actions). Defines what an agent knows.

LAYER 3

Lanes — file scope

Read / write glob patterns declared per role in your project. Defines where an agent may touch — enforced by a branch guard.

Why separate them? Behaviour belongs in personas, knowledge in skills, file scope in your project’s AGENTS.md. Mixing layers is the single most common way teams break the construct. Keep them apart and a role becomes a reusable Lego brick: the same person can play different roles on different work — the role is a function of the work, not the person.
Roles are declared once in a project’s AGENTS.md and validated by a schema. A role with no lane simply cannot start — the construct refuses.

05 • Feature: Personas

Three personas, three jobs

L

Lead

orchestrator • opus

Reads the ticket, decides if design is needed, writes briefs, dispatches tasks, gates every PR, owns ADRs and governance.

Writes no app code. Lane: docs/adr/, AGENTS.md, .ai-dlc.yml, issues. Cannot approve its own PRs.

A

Architect

design • opus

Authors contracts, ADRs and diagrams; refines with the Lead until sign-off. Always runs in plan mode, before any Implementer.

Never implements what it designed. The contract file must land in main before build starts.

I

Implementer

build • sonnet

Reads the brief, works strictly in its lane, contract-first, writes code + tests, ships the PR via finish-feature.

No “while I’m here” refactors. Never edits the Architect’s contract — it halts and flags instead.

Aliases make agents feel like teammates

A role can carry a cosmetic alias (e.g. “Pradhan” for the Lead). It is purely for readability in the hand-off narration — it does not change routing. You get output like:
Pradhan (lead): taking #12 through intake

Model per persona — cost-aware by design

Lead and Architect default to Opus (judgement-heavy). Implementers default to Sonnet (throughput). You can override per role. Expensive thinking is spent where it matters.

04 • Agent Squad × AI-DLC

How the pieces fit together

1. Board manager — the PM layer

ai-dlc-board-manager turns specs into a plan, syncs GitHub issues, plans sprints, and moves tickets between columns. It knows about boards; the construct does not.

2. Agent Squad — the behaviour layer

Personas, role schema, self-review format, lifecycle & tool hooks, and orchestration. This is what makes each agent follow the AI-DLC phases and stay in lane.

3. Your project — the composition

Your repo’s AGENTS.md composes roles from personas + skills + lanes. .ai-dlc.yml wires hooks and the board. One file each, checked in.

The wire between them

A single .ai-dlc.yml at the repo root connects everything:

# .ai-dlc.yml
plan_file: ProjectPlan.md
github_repo: your-org/your-repo
hooks:
  usage_tracker: enabled

When wired, orchestration mirrors each dispatch onto the board. The construct’s own squad board renders regardless — you are never blind.

The guarantee: orchestration changes where the AI-DLC stages run (one session vs. many worktrees), never what they guarantee. Design still precedes build; contracts still merge first; every PR still carries a self-review.

05 • Feature: Orchestration

Orchestration: one Lead, a whole squad

/agent-squad:orchestrate 42 — one session becomes the entire team. The Lead reads the ticket, decomposes it, and visibly hands each task to a named specialist agent, gating every return.

What you see on screen

  • Ticket Intake block — issue, lead, whether design is needed, task list, mode.
  • Squad Board — a live table (Wave / Task / Agent / Lane / Branch / Status), re-rendered on every change.
  • Hand-off block before each agent spawns; Return block when it finishes or halts.

Sequential or parallel — decided automatically

  • Sequential: one branch, one PR, agents run one at a time.
  • Parallel: tasks with non-overlapping lanes run in isolated git worktrees, each with its own branch and PR.
  • Classic shape: DB schema (wave 1) → backend + frontend in parallel (wave 2) → infra (wave 3).

v0.7.0: synchronous dispatch + lost-agent recovery hardened in the field

Synchronous by default. Every agent is spawned and awaited — the Lead never parks work on a detached run it cannot hear back from. “Changes requested” spawns a fresh round-2 agent rather than resuming a finished one.

Never stalls. If a spawn ends with no return, the task is marked halted; the Lead inspects the working tree, salvages committed work, and re-dispatches the rest. A turn never ends silently mid-flight.

05 • Feature: Token estimation

Token estimation: budget, then measure

Every brief carries a coarse size estimate; every session records actual token usage. The gap between them is a calibration loop, not a promise — so estimates get better over time.

The estimate (Lead’s guess, in the brief)

SizeRough total tokens (whole feature)
Sunder ~50k
M~50k – 150k
L~150k – 400k
XLover ~400k

Set as estimate: S|M|L|XL in the brief frontmatter; copied into the session marker at pre-implement.

The actuals (measured, automatically)

A Stop hook reads the session transcript and sums input, output and cache tokens into .agent-squad/usage.json — per issue, per session, never double-counted.

At PR time it surfaces on the pull request:

### Token usage
Estimate: M (~50-150k).
Actual: input=82k output=14k
        cache_read=1.2M
Why management cares: AI coding has a real, variable cost. This turns it into a forecast-and-track line item — you can size features before you start and see the true cost on every PR.

05 • Feature: init

init: adopt the construct in minutes

/agent-squad:init scans a project and generates a draft role composition — so teams onboard without hand-writing schema.

What it scans (read-only)

  • Plan & spec docs: ProjectPlan.md, README.md, specs/, docs/.
  • Tech manifests to detect the stack: package.json (React / Next / TypeScript), pyproject.toml / requirements.txt (Python, FastAPI), Postgres deps, Dockerfile, *.tf, GitHub Actions.
  • Directory shape: frontend/, src/components/ → a frontend lane.

What it generates (a draft to review)

  • AGENTS.md — roles composed from the detected stack: always a Lead, one Implementer per lane, an Architect only if there is a real design surface. Never invents skills that don’t exist.
  • .ai-dlc.yml — the hooks + board wiring file, only if absent.
  • Optionally, a status-line entry so you always see the active persona.
It never overwrites an existing AGENTS.md (it writes AGENTS.md.draft instead) and it does not create plan documents — that stays with the board manager. The output is a starting point a human signs off, then validates with validate-role-schema.

05 • Feature: Governance

Self-review & governance: quality you can audit

Every PR carries a self-review block

Validated by the pre-pr hook — placeholder text is rejected. For an Implementer it must include:

  • What I actively checked (≥3 items)
  • Testable-check verification (command + output + result)
  • How to test, files changed, tests written
  • Issues I found and chose not to fix

It forces the author to re-read its own diff — it is a trace, not a rubber stamp.

22 non-negotiable governance rules

  • ✓ No commits to main; rebase, don’t merge; one feature, one branch.
  • ✓ Stay in your write lane; never overwrite another agent’s files.
  • ✓ Contract-first: API / schema goes in before implementation.
  • ✓ Every PR has a substantive self-review; never approve your own.
  • ✓ No secrets in the repo; no PII in test fixtures.

Rules map to enforcing hooks: branch-guard, rebase-guard, pre-implement, pre-pr. Breaking one blocks the action.

The net effect: the same guardrails a senior reviewer would insist on — lane discipline, contracts, tests, written review — are applied to every agent, on every change, automatically.

06 • Worked example

Example: adding a feature to an open-source FHIR service

Scenario: you fork an open-source healthcare API (say a FHIR validator / query service on GitHub) and need to add a “bulk validate” endpoint — a new API, a DB table, and a small React panel. Perfect for a squad.

1

Lead intake

Reads issue #42. Design needed → yes.
→
2

Architect

Writes the API + schema contract. Merges to main.
→
3

Wave: parallel

backend-dev + frontend-dev in separate worktrees.
→
4

Review + merge

Self-review, Lead gate, merge in order.

The whole thing, one command

# clone the OSS repo, then from its root:
/agent-squad:init            # draft AGENTS.md + .ai-dlc.yml
/agent-squad:orchestrate 42  # Lead runs the squad

The Lead decomposes #42, spawns the Architect, waits for the contract to merge, then fans out backend + frontend in parallel worktrees — all in one visible session.

What lands at the end

  • An Architect PR with the contract + ADR (merged first).
  • A backend PR (app/api/, tests/) and a frontend PR (src/components/) — each in lane.
  • Each PR carries a self-review block and a token-usage line.
  • Issue #42 moved to PR Review on the board.

Any public GitHub repo with a clear stack works — e.g. a FastAPI + Postgres + React service. init reads the manifests and drafts sensible lanes automatically.

06 • Technical quickstart

Quickstart: zero to first feature

# 1. Install deps in the construct (once)
cd /path/to/agent-squad
npm install

# 2. Install as a Claude Code plugin, in your repo
claude plugin install agent-squad

# 3. Draft your role mapping from the project root
/agent-squad:init
#   fallback: copy examples/AGENTS.md.example -> AGENTS.md

# 4. (recommended) bot identity so you can review agent PRs
# PowerShell:
[Environment]::SetEnvironmentVariable('AGENT_SQUAD_GH_TOKEN','<token>','User')

# 5a. Single-session flow (simplest)
/agent-squad:orchestrate 42

# 5b. Or per-lane flow (manual control)
/ai-dlc-board-manager:start-feature 42
/agent-squad:implement 42-backend
/agent-squad:implement 43-frontend
/agent-squad:finish-feature

Verify the construct itself

node tests/ci-dry-run.js
npm test
npm run validate:role-schema

Know who is active

A status-line command renders the live persona:

# e.g.
[agent-squad] implementer:
backend-dev #42 (sonnet)
Prereqs: Node ≥ 18, a Claude Code host, and a git repo. The only runtime dependency is js-yaml.
