AI-DLC x agent-squad

A governed system for AI-driven development

What / Why / How / Demo

WHAT

What is AI-DLC?

AI-DLC = AI-driven Development Life Cycle. The whole delivery loop — not just the IDE moment — run with AI agents in named roles, under a contract a human Lead can supervise.

Classic SDLC

People plan, design, build, review. AI helps with autocomplete inside one IDE.

AI-DLC

Agents take roles across the loop. A human Lead briefs, gates, and merges. The loop is contract-bound, not vibe-bound.

WHAT

Two plugins, one loop

agent-squad

Behaviour contract for the agent

  • Three personas: lead / architect / implementer
  • Role schema + self-review format
  • Lifecycle + tool hooks (branch-guard, rebase-guard)
  • Org-wide skills (aws, python, react, ...)

ai-dlc-board-manager

PM loop on top of a GitHub Project

  • board-sync — plan → board
  • sprint-plan — dependency-ordered queue
  • start-feature / finish-feature
  • Adjacent: daily SM check, milestone retro

Hard split: board-manager owns the board; agent-squad owns the agent.

WHY

What goes wrong without a contract

  • Agents touch files outside the area they were asked to change
  • Scope creeps mid-task — "while I'm here" refactors land in unrelated PRs
  • API / schema contracts get invented at the keyboard; two devs ship incompatible code
  • "Looks good" rubber-stamp PR reviews; no record of what was actually checked
  • No reproducible audit trail when something breaks in production
  • Every team reinvents conventions; nothing carries across projects
These aren't model failures — they're operating-model failures. The model is fine; the system around it is missing.
WHY

"Just prompt better" doesn't scale

What you get with prompts

  • One person's mental model
  • Re-typed every session
  • Untestable, undiffable
  • Drifts every release

What you get with a construct

  • Org-wide behaviour, versioned
  • Skills compose knowledge once
  • Hooks enforce at the moment of action
  • Self-review is validated, not hoped for

The construct is what makes the agent's behaviour auditable and portable.

WHY

What leadership actually buys

Predictability

Same workflow per team, per language. WIP and queue limits visible to anyone.

Safe parallelism

N agents in parallel iff their write lanes don't overlap. Contract-first prevents incompatible builds.

Audit trail

Every PR has a validated self-review. Every status move is owned by a named skill.

Org leverage

Skills compose knowledge once. Contract is semver-locked — no surprise breakage.

HOW

The thesis

Persona  +  Skills  +  Lanes  =  Role

Persona = HOW

Behaviour contract. Fixed set of three.

Skills = WHAT it knows

Pluggable domain knowledge. Org-wide library.

Lanes = WHERE it writes

File globs, per role in AGENTS.md.

Behaviour in personas. Knowledge in skills. Scope in the project's AGENTS.md.

HOW

The three personas (fixed)

PersonaRoleModeWritesCannot
LeadOrchestratorexecuteBriefs, ADRs, governanceApp code; approve own PR
ArchitectDesignerplanContracts, ADRs, architectureApp code; ship without sign-off
ImplementerBuilderexecuteCode + tests in assigned laneTouch contracts; cross lanes
Persona set is fixed at v0.x. Adding a persona is a major version bump. This is the contract.
HOW

Skills are the extension point

Need an agent that knows AWS, or Postgres, or DICOM, or your in-house auth library? You add a skill, not a persona.

---
name: postgres
version: 0.1.0
construct_version: ">=0.1.0"
description: Postgres schema design + migration patterns
persona_affinity: [architect, implementer]
domain: postgres
owner: platform-team
---

## Architect mode
Authors DDL contracts. Considers FK cycles, partitioning, RLS.

## Implementer mode
Writes alembic migrations with reversible downgrade(). Tests on a
disposable DB. Never DROPs without a paired downgrade.

One skill, two modes, decorates whichever persona is in play.

HOW

Lanes pin the scope

Roles compose persona + skills + write/read globs in the project's AGENTS.md:

roles:
  - name: backend-dev
    persona: implementer
    skills: [python, fastapi, postgres]
    lanes:
      write: [app/api/, app/services/, tests/]
      read:  [docs/contracts/, docs/architecture/]

  - name: db-engineer
    persona: implementer
    skills: [postgres, alembic]
    lanes:
      write: [app/models/, alembic/]
      read:  [docs/contracts/]

Two implementers, no lane overlap → safe to run in parallel.

HOW

The end-to-end loop

PO writes the plan → tech lead handoffs → agent does the build → human merges.

HOW

Four stages, no skips

HOW

Governance - non-negotiable

Branch & lane

  • No commits to main
  • Stay within assigned write lane
  • One feature, one branch
  • Rebase, don't merge

Contract & review

  • API / schema contracts go first
  • Don't invent fields not in the contract
  • Every PR has a self-review block
  • Never approve your own PR

22 rules total. Each has a named enforcement owner (hook or review stage).

HOW

Enforcement - two layers

Tool hooks (real-time)

  • branch-guard on Write/Edit — protected branches + lane discipline
  • rebase-guard on Bash — auto-rebase on commit

Host-specific (Claude Code PreToolUse)

Lifecycle hooks (workflow)

  • pre-implement — brief + contract presence
  • pre-pr — self-review validation
  • post-pr — move issue to PR Review

Host-portable. The floor that always applies.

HOW

Self-review = audit trail

## Agent self-review - backend-dev

### What I actively checked (>= 3 items)
- Re-read app/api/todos.py against the brief
- Confirmed schema fields match docs/contracts/todos.yaml
- Walked the diff for stray prints / debug statements

### Testable Check verification
Command:  pytest tests/test_todos.py -v
Output:   [...] 3 passed in 0.42s
Result:   passed

### Files changed
- app/api/todos.py: POST /todos handler
- tests/test_todos.py: 3 cases

### Issues I found and chose not to fix in this PR
- (none)

Section names are contract. validate-self-review rejects missing sections, < 3 checked items, placeholders, or boilerplate.

HOW

A common misread — addressed head-on

"Let's add a persona for our project"

Tempting, and almost always wrong. Persona names (lead / architect / implementer) are contract-fixed. Adding one is a major version bump and breaks cross-team portability.

Don't

Add a persona to fit your domain (e.g. "qa-engineer", "ml-specialist").

Do

Add a role in your project's AGENTS.md — a composition of an existing persona + skills + lanes.

Or

Add a skill under skills/domain/ for org-wide reuse.

DEMO · step 4

One feature, three commands

In chat

  • "Start feature M1-F1"
  • "Implement it"
  • "Finish the feature"

What happens under the hood

  1. start-feature → branch feature/M1-F1, assign, move to Agent Work. Stops.
  2. /agent-squad:implement → pre-implement hook checks brief + contract, writes .agent-squad/session.yml, writes code in backend-dev lane only, runs tests.
  3. finish-feature → pre-pr validates self-review block, opens PR via gh, moves issue to PR Review.

If the agent tries to edit app/models/, branch-guard blocks the Write tool call before it lands. Lane discipline at edit time.

DEMO · step 5

What the Lead sees

On GitHub

  • Issue moved Agent Work → PR Review
  • PR body has a validated self-review block
  • Diff is bounded to app/api/ + tests/
  • Branch was rebased onto main before each commit

In chat (next morning)

  • "Daily SM check" → report on WIP, stale Agent Work, blocked features, PR queue depth
  • "Plan sprint for M2" → read-only proposal of next features in dependency order
Lead reviews. Lead merges. GitHub auto-closes the issue (Closes #N). Board moves to Done.

Adoption path

  1. Install both plugins via the Cowork plugins panel or Claude Code marketplace.
  2. Auth gh + create the board with the five canonical status columns.
  3. Copy examples/AGENTS.md.example and examples/.ai-dlc.yml.example — edit roles + lanes for your team.
  4. Write ProjectPlan.md; run node bin/plan-lint.js.
  5. Verify with bash tests/ci-dry-run.sh (31 smoke tests + 7 CI stages).
  6. Pilot one milestone: one Lead, two implementers with non-overlapping lanes.
  7. Run the retro prompt; feed lessons back into AGENTS.md.

Both plugins are pre-1.0; contract may evolve in 0.x. Locks at v1.0.

The bet

AI agents are useful when they play roles, not when they "complete tasks".

Personas give us behaviour. Skills give us knowledge. Lanes give us scope.
Together they give us a delivery system a human Lead can supervise at scale.
